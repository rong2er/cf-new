# Cloudflare Worker 节点路径 `/?ed=2048` 修改说明文档

本文档旨在详细记录为了让生成的 VLESS/Trojan 节点默认使用 `/?ed=2048` 路径，我们对原始代码进行了哪些必要的修改，以及为什么要这么改。

---

## 1. 为什么单纯修改 `wsPath` 不生效？

在原始代码中，最直观的修改方式是将基础路径变量从 `/` 改为 `/?ed=2048`。
代码中共有 6 处核心定义（例如第 1778、5659 行等）：
```javascript
// 修改前
const wsPath = '/';

// 修改后
const wsPath = '/?ed=2048';
```

**问题出现**：
当你把这行代码改完后，部署到 Worker 并将节点导入到代理客户端（如 v2rayN、Clash）时，你会发现客户端显示的节点路径仍然只有 `/`，后面的 `?ed=2048` 离奇消失了。

**根本原因**：
代理客户端在解析 `vless://` 或 `trojan://` 的 URL 链接时，遇到 `?` 会将其视为“URL 查询参数的开始”。
由于 `/?ed=2048` 中本身包含了特殊字符 `?` 和 `=`，如果不经过 URL 编码直接拼接到链接中（如 `&path=/?ed=2048`），客户端的解析器会发生混乱，直接截断 `?` 后面的内容，导致路径被错误地还原成了 `/`。

---

## 2. 深度修复：强制 URL 转码 (encodeURIComponent)

为了让客户端能够原封不动地收到 `/?ed=2048`，我们必须在代码生成最终订阅链接时，对这个路径进行**强制 URL 转码**。

我们对比了 `原始代码.txt` 和最终的 `明文源吗.txt`，核心修改在于处理**新 IP 节点生成逻辑**（`generateLinksFromNewIPs` 和 `generateTrojanLinksFromNewIPs` 等函数）中的字符串拼接：

### 修改点一：VLESS 节点链接生成 (约 5678 行附近)

```javascript
// 【原始代码】：直接拼接未编码的 wsPath
let link = `${proto}://${user}@${safeIP}:${port}?encryption=none&security=tls&sni=${workerDomain}&fp=${enableECH ? 'chrome' : 'randomized'}&type=ws&host=${workerDomain}&path=${wsPath}`;

// 【修改后】：套上一层 encodeURIComponent 进行安全转码
let link = `${proto}://${user}@${safeIP}:${port}?encryption=none&security=tls&sni=${workerDomain}&fp=${enableECH ? 'chrome' : 'randomized'}&type=ws&host=${workerDomain}&path=${encodeURIComponent(wsPath)}`;
```
*备注：代码中的非 TLS 生成逻辑（约 5693 行）也进行了同样的转码处理。*

### 修改点二：Trojan 节点链接生成 (约 5786 行附近)

```javascript
// 【原始代码】：直接拼接未编码的 wsPath
let link = `${atob('dHJvamFuOi8v')}${password}@${safeIP}:${port}?security=tls&sni=${workerDomain}&fp=chrome&type=ws&host=${workerDomain}&path=${wsPath}`;

// 【修改后】：同样套上 encodeURIComponent
let link = `${atob('dHJvamFuOi8v')}${password}@${safeIP}:${port}?security=tls&sni=${workerDomain}&fp=chrome&type=ws&host=${workerDomain}&path=${encodeURIComponent(wsPath)}`;
```

### 修改点三：兜底错误链接修复 (第 1715 行)

当所有节点获取失败时，系统会生成一个错误提示节点，原本它的路径是硬编码的 `%2F`（即 `/` 的编码）。

```javascript
// 【原始代码】：path=%2F
const errorLink = `${proto}://00000000...&path=%2F#${encodeURIComponent(errorRemark)}`;

// 【修改后】：强制改为 /?ed=2048 的编码形式 %2F%3Fed%3D2048
const errorLink = `${proto}://00000000...&path=%2F%3Fed%3D2048#${encodeURIComponent(errorRemark)}`;
```

---

## 3. 最终效果原理说明

经过上述修改后，代码在生成订阅链接时，不会再输出引起歧义的 `&path=/?ed=2048`，而是输出经过安全转码的：
👉 `&path=%2F%3Fed%3D2048`

**解析流程：**
1. Worker 生成安全编码 `%2F%3Fed%3D2048` 并下发订阅。
2. 用户的代理软件（v2rayN / Clash 等）接收到订阅链接。
3. 代理软件的解析器提取 `path` 参数，并自动执行 URL 解码。
4. 代理软件界面成功显示：`路径 (path): /?ed=2048`。

**总结**：为了实现 `/?ed=2048` 的完美下发，单纯修改变量常量是不够的，必须修补所有未转码的字符串插值漏洞。
